

# إصلاح ظهور طلبات بربح صفر في صفحة متابعة الموظفين

## المشكلة المكتشفة

يوجد **10 طلبات** من موظف محذوف (`f10d8ed9-...`) بحالة "تم التسليم" + "مستلمة الفاتورة" لكن **لا يوجد لها سجل ربح نهائياً** في جدول `profits`. 

```text
السبب الجذري:
─────────────────────────────────────────────────
الطلبات وصلت delivery_status=4 قبل ما يكون التريجر
auto_create_profit_record موجود/فعّال.
→ ما أنشأ لها سجل ربح.
→ الفلتر في الواجهة يتحقق:
   profitRecord?.status === 'no_rule_archived'
→ بما أن profitRecord = undefined (ما موجود)
→ الشرط يرجع false → الطلب يظهر!
─────────────────────────────────────────────────
```

## خطة الإصلاح

### 1. إصلاح قاعدة البيانات (Migration)

إنشاء سجلات `no_rule_archived` لكل طلب:
- `delivery_status = '4'` (واصل)
- لا يملك سجل في `profits`
- `created_by` ليس المدير الرئيسي

هذا يعالج الـ 10 طلبات الحالية وأي حالات مشابهة مستقبلية.

### 2. تحصين الفرونت إند (`EmployeeFollowUpPage.jsx`)

إضافة شرط إضافي في فلتر الطلبات:
- إذا الطلب `delivery_status = '4'` + `receipt_received = true` + **لا يوجد سجل ربح** → يُستبعد تلقائياً
- إذا الطلب يملك سجل ربح مع `employee_profit = 0` + `receipt_received = true` → يُستبعد (يذهب لأرشيف التسوية)

### 3. استبعاد الموظف المحذوف من قائمة الموظفين

الموظف `f10d8ed9-...` ما عنده profile → يجب ألا يظهر كموظف. التحقق أن `employees` list تفلتر فقط الموظفين الموجودين في `profiles` بحالة `active`.

## الملفات المتأثرة

| الملف | التعديل |
|-------|---------|
| Migration SQL جديد | إنشاء سجلات `no_rule_archived` للطلبات اليتيمة |
| `src/pages/EmployeeFollowUpPage.jsx` | تحصين فلتر الطلبات ضد الطلبات بدون سجل ربح |

## قانون عام (سيُطبق)

> **أي طلب واصل (`delivery_status=4`) + مستلم الفاتورة (`receipt_received=true`) + ربح الموظف = 0 أو لا يوجد سجل ربح → يُنقل تلقائياً لأرشيف التسوية ولا يظهر في قائمة الطلبات النشطة.**
