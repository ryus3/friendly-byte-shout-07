أفهم المشكلة الآن بشكل أوضح، وفحصت الكود والبيانات الحالية. يوجد سبب واضح لعدم حذف الطلب، وسبب واضح لضعف مزامنة مدن، ونظام إضافة شركة توصيل بدأ لكن لم يكتمل كـ “نظام عالمي” بعد.

## ما الذي اكتشفته حالياً

### 1. الطلب 140002847 لم يُحذف لماذا؟
الطلب موجود عندنا بهذه البيانات:

```text
tracking_number: 140002847
qr_id: 140002847
delivery_partner: alwaseet
delivery_partner_order_id: 140002847
status: delivery
delivery_status: pending
created_by: المدير العام
```

ويوجد سجل محاولات حذف يقول إن النظام حاول يتأكد من الوسيط، لكن الفحص انحظر بسبب خطأ API:

```text
API errors — refused to delete
checks=0 أو checks=1
initialError=true
```

يعني النظام لم يحذف لأنه عندنا حماية تمنع حذف الطلب الخارجي إذا لم يأتِ تأكيد نظيف من شركة التوصيل أن الطلب غير موجود. هذا صحيح من ناحية الأمان، لكنه حالياً صار صارم أكثر من اللازم بسبب endpoint الوسيط `get-orders-by-ids-bulk` يرجع أحياناً errNum:21 أو خطأ صلاحية أثناء الفحص، فيُفسَّر كـ “غير مؤكد”، فيُمنع الحذف.

المشكلة ليست في زر الحذف فقط، بل في طريقة التحقق من الطلب المحذوف خارجياً.

### 2. لماذا يظهر خطأ عند دخول صفحة الطلبات؟
صفحة الطلبات تشغّل مزامنة تلقائية عند الدخول. هذه المزامنة تفحص الطلبات الظاهرة. عندما تصل للطلب 140002847، تحاول التأكد من الوسيط، فتواجه خطأ API/صلاحية، ثم تمنع الحذف وتظهر رسائل/سلوك مزعج.

إذن الطلب نفسه أصبح “عالقاً”: محذوف من الوسيط، لكن نظامنا لا يملك مسار تحقق كافٍ لتأكيد الحذف بسبب فشل endpoint محدد.

### 3. مزامنة مدن لا تعمل بشكل كافٍ لماذا؟
هناك مشكلتان:

1. `sync-order-updates` موجود ويدعم مدن، لكن لم أجد أي Logs حديثة له، وهذا يعني أنه غالباً لا يعمل فعلياً أو لا يصل له الطلب من الجدولة/الواجهة.
2. منطق حذف الطلبات في الواجهة `performDeletionPassAfterStatusSync` يفحص فقط:

```text
delivery_partner = alwaseet
status = pending
```

لذلك طلبات مدن المحذوفة خارجياً لا تدخل أصلاً في تمرير الحذف التلقائي في الواجهة.

### 4. الطلب السريع: هل الوسيط يجلب المدن والمناطق والأحجام من الكاش؟
الوضع الحالي:

- الوسيط: المدن من `cities_master`، المناطق من `regions_master`، أحجام الطلب من `package_sizes_cache` أو fallback ثابت.
- مدن: المدن من `city_delivery_mappings`، المناطق من `region_delivery_mappings`، أحجام الطلب من `package_sizes_cache`.

الكاش موجود فعلاً:

```text
cities_master: 18
regions_master: 33894
city_delivery_mappings: 36
region_delivery_mappings: 12015
package_sizes_cache: 8
مدن مناطق: 5755
الوسيط مناطق: 6260
```

لكن يوجد خلل محتمل في فلترة مناطق مدن: الكود يفلتر مناطق مدن باستخدام `regions_master.city_id = cityIdForRegions`، بينما `cityIdForRegions` في مدن هو external_id وليس دائماً id الداخلي. حالياً بغداد external_id=1 وcity_id=1 لذلك تعمل، لكن هذا ليس عالمياً وقد يسبب تأخير/فشل لمدن أخرى. الحل الصحيح هو تحويل external city id إلى city_id الداخلي عبر `city_delivery_mappings` ثم فلترة المناطق.

### 5. نظام إضافة شركة توصيل عالمي هل اكتمل؟
لا. الموجود حالياً فقط جدول:

```text
delivery_partners_registry
```

وفيه الوسيط ومدن. لكنه غير مربوط بالكامل بالواجهة والمنطق. النظام لا يزال فيه hardcoding كثير:

```text
['alwaseet', 'modon']
if partner === 'modon'
if partner === 'alwaseet'
```

لذلك حالياً ليس “يدخل كأنه مدن والوسيط بكل مكان” لشركة جديدة. بدأنا الهيكل، لكن يجب إكماله.

## خطة الإصلاح التي سأطبقها بعد الموافقة

### المرحلة 1: إصلاح الطلب العالق والحذف الآمن

1. تعديل منطق التحقق من حذف طلب الوسيط بحيث لا يعتمد على مسار واحد فقط.
2. عند فشل `get-orders-by-ids-bulk` بسبب errNum:21 أو رد غير مؤكد، يستخدم النظام fallback آمن:
   - `merchant-orders` للحساب الصحيح.
   - البحث بـ `tracking_number / qr_id / delivery_partner_order_id`.
   - إذا رجع API بنجاح والطلب غير موجود، نعتبره “محذوف خارجياً”.
3. تعديل حذف الطلب الخارجي في `SuperProvider.jsx` ليستخدم نفس منطق التحقق العالمي، بدل منع الحذف بسبب خطأ endpoint واحد.
4. للطلب 140002847 تحديداً: بعد الإصلاح، إذا أكّد الوسيط أنه غير موجود، سيتم حذفه محلياً أو تعليمه كملغى حسب المنطق الآمن.

### المرحلة 2: إصلاح مزامنة مدن والوسيط من نفس المسار

1. توحيد فحص الطلبات المحذوفة خارجياً لكل الشركاء:

```text
alwaseet
modon
أي شركة مستقبلية لها adapter
```

2. إزالة شرط الواجهة الذي يفحص فقط `delivery_partner = alwaseet`.
3. توسيع شروط الفحص لتشمل الطلبات النشطة/قيد التجهيز/قيد التوصيل، وليس `pending` فقط.
4. جعل `sync-order-updates` يسجل نتيجة تشغيل واضحة في `background_sync_logs` حتى نعرف:
   - كم طلب فحص.
   - كم طلب محدث.
   - كم طلب غير موجود عند الشركة.
   - هل مدن عملت أو لا.
5. استدعاء `sync-order-updates` عند دخول صفحة الطلبات أو زر المزامنة، وليس الاعتماد فقط على مزامنة الواجهة.

### المرحلة 3: إصلاح كاش الطلب السريع بشكل عالمي

1. تعديل جلب مدن ومناطق مدن هكذا:

```text
اختيار مدينة مدن external_id
→ نبحث في city_delivery_mappings عن city_id الداخلي
→ نفلتر region_delivery_mappings حسب regions_master.city_id الداخلي
→ نعرض external_id الخاص بمدن عند الإرسال
```

2. إبقاء fallback API فقط إذا الكاش فارغ فعلاً.
3. توحيد جلب الأحجام من `package_sizes_cache` لكل شركة.
4. إضافة رسالة صغيرة داخل الواجهة عند fallback API حتى نعرف لو الكاش غير مستخدم.

### المرحلة 4: إكمال نظام “شركة توصيل جديدة”

سأكمل النظام على شكل registry + adapter، وليس مجرد زر شكلي.

الهيكل الصحيح:

```text
delivery_partners_registry
  ↓
delivery_partner_tokens
  ↓
Partner Adapter
  login
  createOrder
  editOrder
  deleteOrder/checkDeleted
  listOrders
  cities
  regions
  packageSizes
  statusMap
  ↓
Quick Order / Orders Page / AI Orders / Sync / Invoices
```

سأضيف في نافذة إدارة شركة التوصيل:

1. تبويب “إضافة شركة توصيل” للمدير فقط.
2. حقول واضحة:
   - اسم الشركة بالعربي.
   - اسم الشركة بالإنكليزي.
   - مفتاح الشركة مثل `new_partner`.
   - Base URL.
   - نوع التوثيق: login/token/query/header.
   - endpoints: login/create/edit/list/delete/cities/regions/sizes.
   - خريطة الحالات.
3. شرح داخل الواجهة يوضح كيف تضيف شركة:
   - تضيف بيانات الشركة.
   - تسجل دخول حساب الشركة.
   - تعمل مزامنة مدن/مناطق/أحجام.
   - تعمل اختبار إنشاء طلب تجريبي.
   - بعدها تظهر في الطلب السريع والطلبات والمزامنة.

### المرحلة 5: حدود “العالمي” بصراحة

النظام سيكون عالمياً من ناحية البنية، لكن يجب الانتباه:

- إذا شركة التوصيل الجديدة API مشابهة للوسيط/مدن، يمكن ربطها من الواجهة غالباً.
- إذا API مختلف جداً أو يحتاج توقيع خاص/HMAC/OAuth أو شكل طلب مختلف جداً، نحتاج إضافة adapter صغير بالكود مرة واحدة.
- الهدف الصحيح: الواجهة تضيف 80% من الشركات، والكود يتدخل فقط عند اختلاف بروتوكول جذري.
