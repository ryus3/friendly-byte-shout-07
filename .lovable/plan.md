النتيجة بعد الفحص بدون افتراضات:

1. المشكلة ليست مثبتة كـ “IP whitelist/WAF” في الفحص الحالي.
   - الدالة وصلت إلى واجهة الوسيط ورجعت استجابة JSON من الوسيط نفسه: errNum:21 على get_merchant_invoices لحساب alshmry94.
   - هذا ليس نفس حظر Cloudflare أو مشكلة IP؛ لو كانت IP/WAF لظهر 403/503 أو صفحة Cloudflare/CF_BLOCKED.
   - لكن الكود الحالي يتعامل مع errNum:21 كأنه “لا توجد فواتير”، وهذا خطأ خطير لأنه يخفي سبب المشكلة ويمسح العرض مؤقتاً.

2. الفاتورة الجديدة موجودة في قاعدة البيانات لكن تفاصيل طلباتها غير محفوظة:
   - الفاتورة 3279414 موجودة، المبلغ 292,000، عدد الطلبات 12، لكن delivery_invoice_orders = 0.
   - الفاتورة 3271348 موجودة، عدد الطلبات 5، لكن delivery_invoice_orders = 0.
   - لذلك النافذة تعرض “لا توجد طلبات” لأن جدول تفاصيل الطلبات فارغ فعلاً.

3. سبب فراغ تبويب الفواتير عند الضغط على تحديث واضح:
   - زر التحديث يستدعي API مباشرة.
   - إذا رجع الوسيط errNum:21 أو []، الكود يستبدل قائمة الفواتير الحالية بقائمة فارغة بدلاً من إبقاء بيانات قاعدة البيانات.
   - بعدها عند تحميل الكاش/مزامنة لاحقة ترجع الفواتير للظهور، لذلك تراها فارغة ثم تظهر.

4. المزامنة الخلفية ليست مستقرة كما يجب:
   - cron الساعة 06:00 سجل Timeout بعد 5 ثوانٍ في net._http_response.
   - smart-invoice-sync سجل نجاحاً لكن invoices_synced = 0 بسبب تحويل errNum:21 إلى “لا فواتير”.
   - هذا يعني أن النظام لا يفشل بصوت واضح؛ بل يعتبر الفشل نجاحاً فارغاً، وهذا هو الخلل الجذري.

5. تغييرات آخر 5 أيام التي أثرت على السلوك:
   - تم تعديل alwaseet-api.js و alwaseet-proxy و smart-invoice-sync عدة مرات.
   - التغيير الأخطر: تحويل errNum:21 في endpoints الفواتير إلى [] / “لا توجد فواتير”. هذا أخفى المشكلة الحقيقية وسمح بتفريغ الواجهة.
   - كذلك تم إضافة direct fallback في بعض المسارات، وهذا قد يتجاوز مسار الـ static proxy/الـ IP المضاف للوايت ليست.

خطة الإصلاح الجذري بعد موافقتك:

1. منع تفريغ واجهة الفواتير نهائياً
   - تعديل useAlWaseetInvoices.fetchInvoices ليكون cache-first دائماً.
   - عند الضغط على تحديث: يعرض فواتير قاعدة البيانات فوراً ولا يستبدلها بـ [] إذا فشل API أو رجع errNum:21.
   - إذا رجعت API فارغة، نحتفظ بالقائمة الحالية ونظهر حالة مزامنة واضحة بدلاً من “لا توجد فواتير”.

2. إيقاف تحويل errNum:21 إلى “لا فواتير” بشكل أعمى
   - في alwaseet-api.js: getMerchantInvoices و getInvoiceOrders لن يرجعا [] بصمت عند errNum:21.
   - سنرجع نتيجة منظمة تميز بين:
     - no_data الحقيقي
     - permission_denied من الوسيط
     - rate_limited
     - cloudflare_blocked
     - token_expired الحقيقي
   - هذا يمنع إخفاء الخلل ويمنع مسح الفواتير.

3. تثبيت مسار الـ whitelist/IP
   - في smart-invoice-sync و alwaseet-proxy: الاعتماد على api.ryusbrand.com كمسار أساسي للوسيط.
   - منع fallback المباشر إلى api.alwaseet-iq.net في مزامنة الفواتير إلا كتشخيص واضح وموسوم، لأن الـ direct قد لا يكون ضمن الـ whitelist.
   - إضافة logging يوضح source=static_proxy أو source=direct و upstreamStatus و errNum لكل endpoint.

4. إصلاح طلبات الفاتورة الناقصة
   - تعديل getInvoiceOrders لتطبيع كل أشكال الاستجابة:
     - data.orders
     - orders
     - data كمصفوفة
     - response كمصفوفة
   - عند فتح فاتورة orders_count > 0 و delivery_invoice_orders = 0:
     - استخدام توكن صاحب الفاتورة الصحيح owner_user_id + partner.
     - جلب الطلبات من الوسيط.
     - حفظها فوراً في delivery_invoice_orders.
     - تشغيل link_invoice_orders_to_orders.
   - لو رجع errNum:21، لا نعرض “لا توجد طلبات” كحقيقة؛ نعرض “تعذر جلب تفاصيل الطلبات من الوسيط بسبب صلاحية endpoint/التوكن” ونترك الفاتورة محفوظة.

5. إصلاح المزامنة الخلفية والـ cron
   - جعل comprehensive sync لا يعمل كعملية ضخمة طويلة داخل طلب واحد يضرب timeout.
   - تقسيمها إلى batches لكل توكن/حساب مع حدود واضحة.
   - تعديل cron ليستدعي وضعاً خفيفاً أو يرفع timeout_milliseconds إن كان متاحاً في pg_net.
   - جعل السجل يعتبر errNum:21 فشل/تحذير قابل للتشخيص وليس success=0 صامت.

6. إصلاح مزامنة المدير ومدير القسم
   - الحفاظ على owner_user_id للفواتير والطلبات.
   - المدير يرى فواتير الموظفين من قاعدة البيانات، ولا يحتاج توكنه الشخصي لعرض تفاصيل فاتورة موظف.
   - عند الحاجة لجلب تفاصيل فاتورة موظف، يستخدم النظام توكن نفس صاحب الفاتورة وليس activePartner أو توكن المدير.

7. إصلاح self-healing بدون ترقيع مالي خاطئ
   - إن وجدت طلبات محلية تحمل delivery_partner_invoice_id، نربطها عبر create_invoice_orders_from_local_orders.
   - إن لم توجد محلياً، لا نخترع روابط؛ ننتظر جلب API الصحيح من الوسيط.
   - لا نغيّر receipt_received أو أرباح أو حالة الطلب إلا بعد وجود ربط صحيح، التزاماً بقاعدة عدم الأخطاء.

8. التحقق النهائي بعد التنفيذ
   - اختبار مباشر للفواتير:
     - 3279414 يجب أن تصبح delivery_invoice_orders = 12 أو يظهر سبب API الصريح إن منع الوسيط التفاصيل.
     - 3271348 يجب أن تصبح delivery_invoice_orders = 5 أو يظهر سبب API الصريح.
   - اختبار زر تحديث تبويب الفواتير: لا يجب أن يتحول إلى 0 فواتير أبداً بسبب فشل API.
   - اختبار smart-invoice-sync لحساب alshmry94 ومراجعة logs.
   - اختبار cron/الخلفية حتى لا تسجل timeout صامت.
   - التحقق أن التوكنات بقيت is_active=true ولم يتم تعطيلها تلقائياً بسبب errNum:21.

الخلاصة: الحل ليس رجوع نسخة. الحل هو منع المسح الصامت، إظهار سبب الوسيط الحقيقي، تثبيت مسار الـ whitelist، وتصحيح جلب/حفظ تفاصيل طلبات الفاتورة من توكن صاحب الفاتورة الصحيح.
