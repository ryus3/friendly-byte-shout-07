خطة إصلاح نهائي ومترابط للمشاكل الحالية

1) خطأ الفواتير الحالي: "record new has no field items"
- السبب المرجح والقوي جداً: دالة قاعدة البيانات `record_order_revenue_on_receipt` أُعيدت كتابتها لتقرأ `NEW.items`، بينما جدول `orders` لا يحتوي عمود `items`.
- لهذا عند تحديث استلام الفاتورة/الطلب تنكسر سلسلة التريغرز ويظهر الخطأ كأنه من “حفظ الفواتير”.
- التنفيذ:
  - إنشاء Migration تعيد كتابة الدالة لتقرأ المنتجات من `order_items` مع join على `products/product_variants` بدلاً من `NEW.items`.
  - الإبقاء على توجيه الإيراد حسب `owner_user_id` وتقسيمه على عدة ملاك إن كان الطلب مختلطاً.
  - إضافة fallback واضح إذا لم توجد `order_items`.
  - مراجعة تريغرز `delivery_invoices` و`reconcile_invoice_receipts` للتأكد أن المسار لا يستدعي منطقاً خاطئاً على جدول غير مناسب.
- التحقق:
  - حفظ/مزامنة فاتورة.
  - تحويل الطلب إلى `receipt_received=true`.
  - التأكد أن الحركة تذهب للقاصة الصحيحة حسب مالك المنتج.

2) هل تحسن ربط شركة التوصيل يعني أن الخطأ كان منا؟
- الجواب الدقيق: جزئياً نعم وجزئياً لا.
- ما ظهر من الكود: كان لدينا Regression يزيد الطلبات على API ويعتمد على API في أماكن يجب أن تستخدم الكاش، وهذا يرفع احتمال اصطدامنا بـ WAF/Cloudflare.
- لكن هذا لا يعني أن الحظر السابق “لم يكن منهم”؛ بل يعني أن تعديلاتنا خففت الضغط ومنعت تكرار الاصطدام.
- التنفيذ الوقائي النهائي:
  - تثبيت سياسة `cache-first` فعلياً في كل المسارات.
  - الإبقاء على `REQUEST_GAP_MS` و`inflight dedupe`.
  - منع أي شاشة من طلب المدن/المناطق/الأحجام خارج الكاش إلا إذا كان الكاش فارغاً فعلاً.
  - لاحقاً: إبقاء خيار AWS static IP كحل بنيوي نهائي إذا عادت الحساسية.

3) مشكلة كاش المدن/المناطق يظهر 0 مناطق
- السبب واضح من الكود:
  - `useCitiesCache` يحمّل المناطق في `allRegions`.
  - لكن `CitiesCacheManager` ما زال يعرض `regions.length` ويستخدمه أيضاً في الحالة، بينما `regions` لا يتم تعبئته أصلاً في التحميل العام.
- التنفيذ:
  - تعديل `CitiesCacheManager.jsx` ليعتمد على `allRegions.length` بدلاً من `regions.length`.
  - تعديل حالة “محدث/يحتاج تحديث” لتفحص `allRegions.length`.
  - بعد نجاح المزامنة، إعادة `fetchAllRegions()` وليس `fetchCities()` فقط.
  - فصل عداد العرض عن `syncInfo` حتى لا يعرض صفراً بسبب سجل مزامنة فاشل بينما الكاش ممتلئ.
- النتيجة المطلوبة:
  - تظهر المناطق بالعدد الحقيقي دائماً.
  - يبقى التحديث يعمل.
  - لا يعود رقم 0 الكاذب.

4) قواعد أرباح أحمد لا تظهر لسارة
- وجدت مشكلة منطقية مهمة في صفحة الإعدادات:
  - خيار “كل المنتجات” قيمته الحالية `all`.
  - لكن منطق الحفظ يعتبر أي قيمة موجودة = قاعدة `product`، بينما القاعدة العامة يجب أن تُحفظ `rule_type='default'` و`target_id = null` أو `'default'`.
- هذا قد يؤدي إلى قاعدة محفوظة بشكل غير صالح للحساب أو غير متسق مع الجلب.
- التنفيذ:
  - تطبيع قيمة “كل المنتجات” قبل الحفظ: `all -> null`.
  - إجبار القاعدة العامة على `rule_type='default'`.
  - توحيد الجلب والعرض بحيث يتعامل مع `(target_id IS NULL OR target_id='default')`.
  - التأكد أن العرض لا يعتمد على ربط منتج موجود حتى تظهر القاعدة العامة.
  - تحسين refresh الفوري بعد الحفظ/الحذف.
- هذا هو الإصلاح الأهم لهذه الشاشة.

5) لماذا تسجيل الدخول لا يُحفظ؟
- الإعدادات الأساسية صحيحة (`persistSession: true` و`autoRefreshToken: true`).
- لكن توجد نقطة هشة في `UnifiedAuthContext`:
  - `fetchUserProfile` يستخدم `user_roles!inner`، فإذا تعثّر الربط أو لم تُرجع الأدوار كما ينبغي قد يُعامل المستخدم وكأنه بلا بروفايل/غير نشط رغم وجود session محفوظة.
- التنفيذ:
  - فك جلب البروفايل عن جلب الأدوار: جلب `profiles` أولاً بدون `inner join`، ثم جلب الأدوار في استعلام مستقل.
  - إضافة Auth-ready gating صريح حتى لا تبني الواجهة أو الحماية قبل اكتمال استعادة الجلسة.
  - مراجعة أي مسار يعيد `setUser(null)` مبكراً أثناء bootstrap.
- الهدف:
  - إعادة فتح المتصفح = يبقى المستخدم مسجلاً.
  - لا يحدث logout وهمي بسبب race أو join حساس.

6) لماذا إضافة المنتجات/تعديل الإعدادات ليس فورياً؟
- السبب في طبقة البيانات:
  - `SuperAPI.setupRealtimeSubscriptions` يستمع لجدول `products` لكن `handleRealtimeUpdate` لا يحدّث المنتجات مباشرة.
  - `refreshProducts` في `SuperProvider` حالياً يستدعي `fetchAllData()` الكامل، وليس تحديثاً موجهاً للمنتجات فقط.
- التنفيذ:
  - إضافة معالجة مباشرة لجداول `products`, `product_variants`, وربما `employee_profit_rules`, `settings`.
  - عند تغير منتج: استدعاء تحديث منتجات موجّه وليس `fetchAllData` الكامل.
  - بعد إضافة/تعديل منتج: استخدام `dbRefetchProducts()` أو دمج مباشر في الحالة.
  - بعد حفظ قاعدة ربح: إعادة جلب القواعد فوراً وتحديث العرض محلياً.
- النتيجة:
  - الإضافة والتعديل يظهران فوراً بدون انتظار أو بطء ملحوظ.

7) التأكد أن النظام يعرف المنتج ومالكه ويوجه الحركة للقاصة الصحيحة
- هذا سيتم تثبيته ضمن إصلاح بند الفواتير نفسه.
- التنفيذ:
  - الاعتماد على `order_items` كمصدر الحقيقة.
  - ربط كل عنصر بمنتجه ثم `owner_user_id`.
  - تجميع المبالغ لكل مالك.
  - إدخال حركة منفصلة لكل قاصة عند الطلبات المختلطة.
  - fallback واضح للطلبات القديمة/الناقصة.
- ثم اختبار 3 حالات:
  - طلب منتج واحد لمالك واحد.
  - طلب مختلط لمالكين.
  - طلب بلا عناصر مكتملة.

8) ما نحتاجه لتحويله إلى تطبيق “حقيقي”
- بعد إصلاح المشاكل الحالية، المرحلة التالية:
  - خادم Proxy ثابت IP على AWS Lightsail/EC2 + Elastic IP لشركات التوصيل.
  - مراقبة أخطاء مركزية (Sentry أو ما شابه) للواجهة + Edge Functions.
  - سجل تشخيص واضح للفواتير والمزامنات والحركات المالية.
  - اختبارات E2E إلزامية للمسارات الحساسة:
    - تسجيل الدخول وحفظ الجلسة
    - مزامنة الفواتير
    - استلام الفاتورة وتوليد الحركة
    - قواعد الربح
    - عرض الكاش
  - مراجعة RLS وسياسات الجداول الحساسة.
  - نسخ احتياطية وخطة rollback للمigrations.
