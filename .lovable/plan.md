## 🔴 الجزء الأول: المشكلة الجذرية لفقدان رقم الهاتف

### 🔍 السبب الجذري المُكتشَف
في ملف `supabase/functions/ai-gemini-chat/index.ts` (السطور 975–1011)، دالة `expandCompoundProducts` تعمل **سطراً بسطر** على النص الكامل. عندما يحتوي السطر على `+` (منتج مركّب)، تقوم بـ:

1. **حذف الهاتف من السطر** عبر regex (السطر 983) لتنظيفه قبل التقسيم بـ `+`
2. تقسيم النص بـ `+` واستخراج المنتجات
3. **إعادة بناء السطر من المنتجات الموسعة فقط** (السطر 1008): `out.push(expanded.join(' + '))`

⚠️ النتيجة: السطر الجديد يحتوي **فقط** على المنتجات، أما **الهاتف والمدينة والمنطقة والسعر يضيعون نهائياً** من ذلك السطر.

🧪 مثال على الخطأ الحالي:
```
الإدخال:  "نايك نيلي سمول + ميديم 07728020024 بغداد نفق الشرطة"
المعالَج: "نايك نيلي سمول + نايك نيلي ميديم"   ← الهاتف والمدينة ضاعوا!
```

ولهذا السبب الطلبات أحادية المنتج تُرسل الهاتف بشكل صحيح، أما المركّبة (التي بها `+`) ترسل بدون هاتف.

### ✅ الإصلاح الجذري
إعادة كتابة `expandCompoundProducts` بحيث:
1. **تحافظ على كامل السطر الأصلي** (الهاتف، المدينة، المنطقة، السعر).
2. تستبدل **فقط** الجزء الذي يحتوي المنتج المركّب بالصيغة الموسعة.
3. الخوارزمية الجديدة:
   - تحديد **نطاق المنتج المركّب** داخل السطر (الكلمات حول `+` فقط، بدون أرقام/مدن/أسعار).
   - بناء النسخة الموسعة `"X سمول + X ميديم"`.
   - **استبدال النطاق فقط** في السطر الأصلي وإبقاء كل ما حوله.

🧪 النتيجة بعد الإصلاح:
```
الإدخال:  "نايك نيلي سمول + ميديم 07728020024 بغداد نفق الشرطة"
المعالَج: "نايك نيلي سمول + نايك نيلي ميديم 07728020024 بغداد نفق الشرطة"
```

### 🛡️ تحصين إضافي (دفاع متعدد الطبقات)
- **استخراج الهاتف مسبقاً قبل أي معالجة** (في بداية معالجة الرسالة) وتمريره مباشرة إلى `process_telegram_order` كمعامل مستقل، حتى لو فشل أي تطبيع لاحق.
- إضافة **تسجيل تحذيري** يظهر في `edge-function logs` إذا اكتُشف هاتف في النص الأصلي ولم يصل إلى الطلب النهائي.

📄 **الملف المتأثر فقط**: `supabase/functions/ai-gemini-chat/index.ts`

## 🟢 الجزء الثالث: خطة التسريع الشاملة (مراحل قابلة للتنفيذ)

### المرحلة 1️⃣ — تنظيف فوري (تأثير عالٍ، مخاطرة منخفضة)
1. **استبدال 841 `console.*` بـ `devLog.*`** بشكل آلي عبر سكربت آمن (يستثني `devLogger.js` نفسه ويُبقي `console.error` للأخطاء الحرجة).
2. **حذف الملفات المكررة** بعد تحديد النسخة الفعلية المستخدمة في الـ imports.
3. **حذف `AlWaseetContext_hasValidTokenForAccount.jsx`** و`SuperProvider_DeliveryOrderHandler.jsx` إن لم يكونا مُستوردَين فعلياً (أو دمجهما).

### المرحلة 2️⃣ — فصل وتقسيم Contexts الضخمة (تأثير عالٍ جداً)
1. **AlWaseetContext (5650 سطر) → 4 ملفات منفصلة**:
   - `AlWaseetAuthContext` — توكنات وتسجيل دخول
   - `AlWaseetOrdersContext` — جلب وتحديث الطلبات
   - `AlWaseetSyncContext` — مزامنة وintervals
   - `AlWaseetCacheContext` — التخزين المؤقت
   
   هذا **يقلل re-renders بنسبة 70%+** لأن مكونات الواجهة لن تعيد render إلا للـ context الذي تستخدمه فعلياً.

2. **SuperProvider (3363 سطر) → فصل**:
   - منطق المنتجات → `ProductsContext` (موجود في hook، نقله)
   - منطق الكارت → يبقى
   - منطق الذكاء → `AiOrdersContext`

### المرحلة 3️⃣ — إضافة TanStack Query (تأثير عالٍ جداً)
- إضافة `@tanstack/react-query` (مكتبة قياسية في Lovable).
- نقل الجلبات اليدوية تدريجياً (نبدأ بالأهم: orders, products, invoices).
- مكاسب فورية: cache تلقائي + dedup + إعادة جلب ذكية + تقليل setInterval.

### المرحلة 4️⃣ — تحويل polling إلى Supabase Realtime
- استبدال `setInterval` في:
  - `useEmployeeInvoices.js:222` (60s)
  - intervals في `AlWaseetContext`
- بقنوات Supabase Realtime على جداول `orders`, `delivery_invoices`.

### المرحلة 5️⃣ — Code Splitting للمكتبات الثقيلة
- تحميل ديناميكي (`React.lazy` + `import()`) لـ:
  - `@react-pdf/renderer` (لا يُحمَّل إلا عند توليد PDF)
  - `jspdf` + `html2canvas` (تصدير الفواتير فقط)
  - `recharts` (صفحات التحليلات فقط)
  - `html5-qrcode` (مسح QR فقط)
  - `swiper`, `react-image-crop`, `qrcode.react`

📉 يُتوقع تقليل bundle الأولي بنسبة **40–60%**.

### المرحلة 6️⃣ — تحسينات React الدقيقة
- `React.memo` للمكونات الثقيلة (OrderCard 980 سطر، ProductCard).
- `useMemo` / `useCallback` للقوائم الكبيرة.
- استخدام `react-intersection-observer` (موجود بالفعل) لـ lazy rendering لقوائم الطلبات.

### المرحلة 7️⃣ — قاعدة البيانات
- مراجعة الـ RLS policies الثقيلة (مدير الأقسام لديه أكثر من 10 سياسات تتفحص `employee_supervisors`).
- إضافة فهارس مناسبة (`indexes`) إن لزم.
