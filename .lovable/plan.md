
## ما حصل فعلياً للطلب 3246226 (مدن)

فحصت قاعدة البيانات وسجلات المزامنة. هذا ما حدث بالضبط:

```text
الطلب 3246226 (مدن)
  delivery_partner: modon
  partner_missed_count: 2  ← وصل للضربتين
  delivery_status: 32 (رفض/ملغى من الشريك)
  status: cancelled
  آخر تحديث: 16:53 (نفس وقت آخر سجل sync-order-updates)
```

سجلات المزامنة تؤكد أن `sync-order-updates` اشتغل وفحص `[alwaseet, modon]` ونجح بتحديث طلب واحد. أي **منطق "الضربتين" يعمل بشكل صحيح**: الطلب اختفى من API مدن مرتين متتاليتين، فعلّمه النظام كملغى من الشريك (32) وحرّر المخزون عبر الـ trigger.

### إذن لماذا يظهر "ملغى" بدل أن يُحذف؟

لأن المنطق الحالي مقصود كـ **حذف ناعم**:
- يُبقي السجل في قاعدة البيانات للسجل التاريخي/المحاسبي
- يحرّر المخزون المحجوز
- يحوّل الحالة إلى ملغى ليُستبعد من العمليات المستقبلية

لكنك تتوقع **حذف فعلي** (يختفي من قائمة الطلبات نهائياً). المشكلة في القرار، ليس في الكود.

## التعديل المقترح — حذف فعلي آمن عند تأكد الحذف الخارجي

عندما تكتشف المزامنة أن الطلب محذوف فعلاً من شركة التوصيل (الضربتان)، نطبّق هذا المسار:

```text
1. إذا الطلب لا يملك إيصال مستلم (receipt_received != true)
   ولا فاتورة مرتبطة
   ولا أرباح مدفوعة لموظف
   →  حذف فعلي من جدول orders (الـ trigger يحرّر المخزون تلقائياً)

2. خلاف ذلك (يوجد ارتباط محاسبي)
   →  إبقاء الـ "حذف الناعم" الحالي: status=cancelled, delivery_status=32
       لحماية السجلات المالية
```

هذا يحقق:
- الطلب 3246226 الذي ليس له ارتباط مالي → سيُحذف فعلياً عند المزامنة القادمة
- الطلبات التي تأثرت محاسبياً → تبقى محفوظة كملغاة (حماية البيانات)

### تنظيف الطلب الحالي 3246226

سيتم حذفه فعلياً ضمن نفس التشغيل الأول للمزامنة بعد التعديل (لأنه ليس له ارتباطات).

## إضافة شركة توصيل جديدة — الواقع الكامل

النموذج الحالي يطلب فقط: مفتاح + اسم + Base URL. **هذا غير كافٍ إطلاقاً** لشركة حقيقية. كل شركة توصيل لها تفاصيل مختلفة جوهرياً، وأنت محق أن وثائق الوسيط مثلاً (الصورة التي أرسلتها) تحتوي 9 أقسام كاملة.

### ما يحتاجه النظام فعلاً لربط شركة جديدة

بناءً على فحص الوسيط ومدن، أي شركة تحتاج المعلومات التالية:

```text
الهوية:
  - مفتاح فريد (alwaseet, modon, new_partner)
  - اسم عربي/إنكليزي
  - الشعار (اختياري)

الشبكة:
  - Base URL  (مثال مدن: https://mcht.modon-express.net/v1/merchant)
  - هل يحتاج Proxy وسيط؟ (الوسيط نعم بسبب WAF، مدن لا)
  - رابط Proxy (إن وُجد)

التوثيق:
  - نوع التوثيق: token/api_key/oauth
  - مكان التوكن: query string (?token=) أو header (Authorization)
  - مدة صلاحية التوكن (للتجديد التلقائي)

نقاط النهاية (Endpoints) — لكل عملية:
  - login          (POST مع username/password)
  - createOrder    (POST مع بيانات الطلب)
  - editOrder      (POST/PUT)
  - listOrders     (GET — لجلب طلبات التاجر)
  - statusList     (GET — قائمة الحالات الممكنة)
  - bulkByIds      (GET — جلب طلبات محددة بمعرفاتها)
  - cities         (GET — قائمة المدن)
  - regions        (GET — قائمة المناطق لمدينة)
  - packageSizes   (GET — أحجام الطلب)
  - invoices       (GET — قائمة الفواتير)
  - invoiceOrders  (GET — طلبات فاتورة محددة)

شكل البيانات (مهم جداً ويختلف بين الشركات):
  - أسماء حقول إنشاء الطلب
    (مثلاً: client_name vs customer_name vs name)
  - شكل العنوان (city_id منفصل، أو نص حر)
  - حقل السعر (price، total، amount)
  - حقل الكمية وأحجام السلع
  - شكل رد API (data في غلاف، أو مباشرة)

خريطة الحالات (Status Mapping) — حرجة:
  - كل شركة لها أرقام حالات مختلفة
  - يجب ربطها بحالاتنا الداخلية:
    1=قيد التجهيز، 4=مُسلَّم، 17=مُرتجع، 32=ملغى ...
```

### الخلاصة الواقعية

> **هل يكفي زر في الواجهة لإضافة شركة جديدة؟ لا.**
> **هل يحتاج برمجة كاملة كل مرة؟ ليس بالضرورة.**

الحل الواقعي العالمي = نظام **Registry + Adapter** على مستويين:

```text
المستوى 1 — Registry (من الواجهة، بدون كود):
  ✓ معلومات الهوية والشبكة والـ endpoints وخريطة الحالات
  ✓ شكل بيانات الطلب الأساسي (أسماء الحقول)
  ✓ قائمة المدن والمناطق (تُجلب وتُخزّن تلقائياً بعد تسجيل الدخول)
  → يكفي للشركات التي تتبع نفس النمط (REST + token + JSON بسيط)

المستوى 2 — Adapter بالكود (مرة واحدة لكل شركة):
  ✗ توقيع HMAC أو OAuth معقد
  ✗ شكل بيانات غير قياسي (XML, multipart غريب)
  ✗ منطق خاص لتحويل العناوين أو الأحجام
  → يحتاج ملف JS صغير في src/lib/adapters/
```

## ما سأنفّذه بعد الموافقة

### 1. إصلاح الحذف الفعلي في `sync-order-updates`

تعديل المنطق عند الضربة الثانية:
- فحص: هل للطلب إيصال مستلم/فاتورة/أرباح موظف مدفوعة؟
- لا → حذف فعلي من جدول `orders`
- نعم → إبقاء الحذف الناعم الحالي

نتيجة فورية: الطلب 3246226 سيُحذف فعلياً عند أول مزامنة قادمة.

### 2. تطوير نموذج "إضافة شركة توصيل" ليكون حقيقياً

استبدال النموذج الحالي (3 حقول) بنموذج متعدد الأقسام:

```text
القسم 1: الهوية         (مفتاح، اسم عربي/إنكليزي)
القسم 2: الشبكة          (Base URL، Proxy URL اختياري)
القسم 3: التوثيق         (نوع، مكان التوكن، مدة الصلاحية)
القسم 4: نقاط النهاية   (10 endpoints قابلة للتعديل، مع قيم افتراضية ذكية)
القسم 5: خريطة الحالات (جدول: رقم الشركة → حالتنا الداخلية)
القسم 6: شكل بيانات الطلب (تطابق الحقول)
```

كل قسم قابل للطي، مع قيم افتراضية مأخوذة من الوسيط/مدن، وأمثلة جنب كل حقل.

### 3. شرح داخل الواجهة (Documentation Panel)

إضافة قسم "كيف أضيف شركة" يشرح بالعربي بمثال كامل من الوسيط ومدن:
- كل حقل ولماذا
- من أين تجد القيمة في وثائق الشركة
- متى تحتاج Adapter بالكود (وكيف تطلبه مني)

### 4. تحذير صريح في الواجهة

> "إذا API الشركة الجديدة يستخدم نمطاً غير قياسي (HMAC, OAuth, XML)، تواصل مع المطور لإضافة Adapter. هذا النموذج يكفي لـ ~80% من شركات التوصيل العراقية."

## نقاط أؤكدها بصراحة

1. **الطلب 3246226 لم يُحذف** لأن النظام عامداً يحوّله لـ "ملغى" — سأغيّر هذا.
2. **النموذج الحالي لإضافة شركة لا يكفي** — أنت محق تماماً. سأوسّعه ليطابق الواقع.
3. **لن يكون نظاماً سحرياً 100%**: ~80% من الشركات يكفيها النموذج، الباقي يحتاج Adapter صغير بالكود (أنا أكتبه عند الحاجة).
4. لن أعد بأن أي شركة في العالم ستُربط بزر فقط — هذا غير صحيح تقنياً ولن أكذب عليك.
