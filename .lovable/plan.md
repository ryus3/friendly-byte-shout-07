## ما المشكلة فعلاً؟

- الحذف التلقائي يعمل بالخلفية لأن Edge Function تستخدم عداد `partner_missed_count` وتاريخ 3 دقائق وتتحقق من الشريك بشكل مستقل.
- في المزامنة داخل الموقع يوجد أكثر من مسار حذف قديم في `AlWaseetContext.jsx`، وبعضها لا يستخدم نفس حماية الخلفية، وبعضها يستخدم توكن السياق الحالي بدل توكن الحساب الذي أُنشئ به الطلب. لذلك النتيجة غير ثابتة.
- ترتيب الطلبات يتأثر لأن بعض مسارات المزامنة تحدّث `updated_at` حتى بدون تغيير حالة، والصفحات تعتمد أحياناً على `updated_at` أو لا ترتب أصلاً.
- الموافقة على الطلب الذكي تفشل لأن المسار الحالي معقد: مطابقة منتجات، حجز مخزون، إنشاء في شركة توصيل، إنشاء محلي، إدراج العناصر، ثم حذف الطلب الذكي. أي فشل في خطوة وسطية قد يترك الطلب بدون رسالة واضحة أو لا ينقص العداد فوراً.
- نافذة المخزون المحجوز تعيد حساب كل الطلبات والعناصر من السياق الكامل عند الفتح، وهذا ثقيل ومتذبذب؛ كما أنها لا تستخدم دائماً البيانات الممررة من صفحة المخزون وتحتوي أنماط CSS مباشرة تحتاج تنظيف.

## خطة الإصلاح العاجل بدون تخريب

### 1. توحيد الحذف التلقائي داخل الموقع مع الخلفية
- إنشاء دالة واحدة داخل `AlWaseetContext.jsx` للتعامل مع “طلب غير موجود لدى الشريك”.
- الحذف يتم فقط عند تحقق كل الشروط معاً:
  - `partner_missed_count >= 3`.
  - عمر الطلب 3 دقائق أو أكثر.
  - `delivery_status` فقط `1` أو `pending`.
  - لا توجد فاتورة ولا استلام وصل.
  - الطلب غير مكتمل/مرتجع/ملغى.
  - الشريك نفسه نجحت مزامنته في هذه الدورة.
  - يوجد توكن صالح لنفس `delivery_account_used` الذي أُنشئ به الطلب.
- أي خطأ API أو Rate limit أو توكن غير صالح لا يحذف الطلب، فقط يتوقف ويظهر سبب واضح.
- إزالة/تعطيل مسارات الحذف القديمة التي تحذف مباشرة عبر `fastSync` أو توكن عام.

### 2. إصلاح الموافقة على الطلبات الذكية وجعلها “كما كانت” وبسيطة
- جعل `approveAiOrder` يعمل كعملية ذرية واضحة:
  1. جلب الطلب الذكي.
  2. مطابقة المنتجات والمتغيرات.
  3. حجز المخزون.
  4. إنشاء الطلب في شركة التوصيل إن كانت الوجهة خارجية.
  5. إنشاء الطلب المحلي وعناصره.
  6. ربط الطلب الذكي بالطلب الحقيقي ثم تحديثه/حذفه بأمان.
- إضافة تراجع آمن: إذا فشل إنشاء الطلب المحلي بعد حجز المخزون، يتم تحرير الحجز فوراً.
- إرجاع سبب فشل واضح دائماً: توكن، موقع، هاتف، مخزون، منتج غير مطابق، API.
- تحديث عداد المحدد فور كل نجاح أو حذف: إذا المحدد 8 ونجح طلب يصبح 7 مباشرة، ولا ينتظر إعادة الجلب.
- عدم وضع الطلب في “approved” أو إخفائه إلا بعد نجاح إنشاء الطلب الحقيقي فعلاً.

### 3. ترتيب الطلبات حسب آخر تغيير حالة حقيقي فقط
- عدم تحديث `updated_at` عندما لا تتغير الحالة أثناء المزامنة المرئية.
- تحديث `status_changed_at` فقط عندما يتغير `delivery_status` أو `status` فعلاً.
- صفحة الطلبات وصفحة متابعة الموظفين ترتب حسب:
  1. `status_changed_at` تنازلياً.
  2. ثم `created_at` تنازلياً.
- الطلب الذي تمت مزامنته بدون تغيير لا يصعد للأعلى.

### 4. إصلاح انهيار نافذة المخزون المحجوز
- جعل `ReservedStockDialog` يستخدم `reservedOrders` و `allUsers` الممررة من `InventoryPage` بدل إعادة قراءة كل شيء من `useSuper` عند الفتح.
- تغليف الحسابات الثقيلة بـ `useMemo` مع guards صارمة للبيانات الناقصة.
- إصلاح مطابقة الموظف عبر `user_id` و `id` معاً.
- إزالة الأنماط المباشرة غير المتوافقة مع نظام التصميم قدر الإمكان.
- إضافة حالة فارغة/تحميل بدل الانهيار عند وجود طلب أو عنصر ناقص البيانات.

### 5. إصلاح خطأ تحميل module script
- فحص سبب `TypeError: Importing a module script failed` بعد الإصلاحات، غالباً ناتج عن import لمسار غير موجود أو chunk فشل تحميله.
- إصلاح المسار/الاستيراد إن ظهر في السجلات بعد تنفيذ التعديلات.

## خطة تطوير المتجر المتكامل

### المرحلة 1: متجر فعلي لكل مستخدم
- كل موظف لديه متجر مستقل عبر slug مثل `shop.ryusbrand.com/{employee}`.
- المنتجات المعروضة فقط التي يسمح بها المدير/مدير القسم عبر صلاحيات المنتجات الحالية.
- الفلترة: القسم، النوع، الموسم، اللون، القياس، السعر، المتوفر فقط.
- صفحات: الرئيسية، المنتجات، تفاصيل المنتج، السلة، إتمام الطلب، تتبع الطلب.

### المرحلة 2: طلبات الزبائن من المتجر
- الزبون يختار المنتج/اللون/القياس والكمية.
- يدخل الاسم، رقم الهاتف، رقم ثاني اختياري، المدينة، المنطقة، أقرب نقطة، ملاحظات.
- الطلب يصل كـ `ai_order` مصدره `store` للموظف صاحب المتجر.
- خيار إرسال الطلب إلى واتساب الموظف كمسار إضافي، لكن الاعتماد الأساسي يكون داخل النظام حتى لا تضيع الطلبات.

### المرحلة 3: واجهة احترافية عالمية
- تصميم قريب من الصور المرفقة: متجر داكن، بطاقات منتجات، قصص/Reels، بحث ذكي، فلاتر عائمة، تجربة موبايل قوية.
- استخدام صور المنتجات الحقيقية من النظام أولاً.
- Reels داخلية: رفع فيديو قصير وربطه بمنتج واحد أو عدة منتجات، مع زر شراء مباشر.

### المرحلة 4: تتبع الشحنة للزبون
- صفحة تتبع عامة برقم الهاتف أو رقم التتبع.
- تعرض مراحل الطلب من بيانات النظام وشركة التوصيل.
- لا تكشف بيانات حساسة، فقط حالة الطلب والموعد التقريبي والمعلومات الأساسية.

### المرحلة 5: دعم ذكي حقيقي
- مساعد ذكي مرتبط ببيانات المتجر والطلبات المسموح عرضها للزبون.
- يجيب عن: أين طلبي؟ المقاسات، سياسة الإرجاع، المنتجات المتوفرة.
- أي إجراء حساس مثل تعديل طلب أو إلغاء يرسل طلب مراجعة للموظف ولا ينفذه تلقائياً.

### المرحلة 6: الواقع المعزز AR
- ممكن، لكنه يعتمد على توفر ملفات 3D للمنتجات `glb/usdz`.
- للملابس: AR الحقيقي أصعب ويحتاج نماذج أو تكامل قياس/تجربة افتراضية.
- للأحذية/الحقائب/الإكسسوارات أسهل: عرض 3D عبر `<model-viewer>` وزر “عرض في الواقع المعزز”.
- نبدأ بمنتجات تدعم 3D اختيارياً ولا نجبر كل المنتجات.

## التنفيذ المقترح الآن

أبدأ بالإصلاحات العاجلة أولاً فقط:
1. الحذف التلقائي.
2. الموافقة والعداد في الطلبات الذكية.
3. ترتيب الطلبات حسب تغيير الحالة.
4. نافذة المخزون المحجوز.

ثم بعد استقرارها نفتح خطة المتجر كمرحلة مستقلة حتى لا نخلط إصلاحات حساسة مع بناء كبير.
