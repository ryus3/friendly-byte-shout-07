
## التشخيص

نعم، كلامك صحيح 100%. النظام يدعم الفلترة فعلياً (المعامل `viewerUserId` موجود في `linkInvoiceWithLocalOrders`)، لكنه **غير مُمرَّر** في جميع نقاط الفتح:

| المكان | الحالة الحالية | المطلوب |
|---|---|---|
| `AlWaseetInvoicesTab` (متابعة الموظفين › تبويب الفواتير — يفتحها الموظف لنفسه أو المدير لنفسه) | `viewerUserId` غير مُمرَّر → يرجع **كل** الطلبات المرتبطة بالفاتورة (طلبات المدير + أحمد + أي شخص آخر يشارك حساب `alshmry94`) | تمرير `user.id` الحالي → كل مستخدم يرى طلباته فقط |
| `EmployeeDeliveryInvoicesTab` (متابعة الموظفين › فواتير موظف معيّن) | يمرّر `employeeId` عند اختيار موظف ✅ | لا تغيير |
| `AllEmployeesInvoicesView` (نظرة عامة للمدير) | يمرّر `employeeFilter` عند اختيار موظف ✅، و `null` عند "الكل" (المدير يرى الجميع) | لا تغيير |

نتيجة الإصلاح:
- **أحمد** يفتح فاتورة `3406747` → يرى فقط طلباته المحلية المرتبطة (إن وُجدت).
- **المدير العام** يفتح نفس الفاتورة من صفحته → يرى طلباته الخاصة فقط.
- **المدير العام** يفتحها من صفحة متابعة الموظف "أحمد" → يرى طلبات أحمد فقط (يعمل أصلاً).

لا مساس بالمزامنة، ولا بـ RLS على جدول الفواتير، ولا بأي منطق مالي — الفلترة تطبَّق في طبقة التطبيق على نتيجة الربط فقط.

## لماذا تُعاد محاولة الجلب؟

السبب الحقيقي: في `AlWaseetInvoiceDetailsDialog` كل فتحة للحوار:
1. تستدعي `fetchInvoiceOrders` بدون `preferCache`.
2. تستدعي `loadLinkedOrders` التي تعيد البحث في القاعدة.

داخل `fetchInvoiceOrders` يوجد بالفعل شرط ذكي (`cacheOnly` عندما الفاتورة مستلمة + الكاش مكتمل)، لذا الفاتورة `3406747` (52/49) **لا تضرب API فعلياً** — لكن لحظة الفتح يُعاد ضبط `state` فيظهر هيكل تحميل (skeleton) قصير يوحي بإعادة الجلب. أمّا فاتورة `1849184` فعدّاد الكاش = 0/1 → الشرط يقرر محاولة API، وبسبب اختلاف توكن الحساب أو تأخر شركة التوصيل تفشل المحاولة وتظهر رسالة "تعذّر الجلب".

الحل:
1. **تخزين مؤقت داخل الجلسة** لقوائم `invoiceOrders` و `linkedOrders` لكل `(invoiceId, viewerUserId)` داخل الحوار، فلا يظهر skeleton عند إعادة الفتح إن كانت البيانات محمّلة سابقاً.
2. تمرير `preferCache: true` صراحةً عند فتح فاتورة مستلمة + مكتملة الكاش (تأكيد سلوك "محفوظة، لا تُسأل API").
3. للفواتير غير المكتملة (مثل 1849184): إبقاء المحاولة الصامتة في الخلفية مع رسالة "محفوظة وسيُعاد المحاولة" كما هي الآن.

## الخطة التنفيذية

### 1) `src/components/orders/AlWaseetInvoicesTab.jsx`
- استيراد `useAuth` وتمرير `viewerUserId={user?.user_id || user?.id}` إلى `<AlWaseetInvoiceDetailsDialog>`.

### 2) `src/components/orders/AlWaseetInvoiceDetailsDialog.jsx`
- إضافة `useRef` لخريطة كاش داخل الجلسة: `cacheRef.current[invoiceId + '|' + (viewerUserId||'self')] = { orders, linked }`.
- في `useEffect`: إذا توفر كاش لنفس المفتاح → استخدمه فوراً ولا تستدعِ `fetchInvoiceOrders`/`loadLinkedOrders` إلا في الخلفية بصمت لتحديث الأرقام.
- تمرير `{ preferCache: true }` إلى `fetchInvoiceOrders` عندما تكون الفاتورة `isReceived` (مرّة واحدة كافية).
- إبقاء رسالة "تعذّر جلب التفاصيل" فقط للحالات الفعلية (cached=0 و expected>0).

### 3) لا تغييرات في:
- `EmployeeDeliveryInvoicesTab.jsx`، `AllEmployeesInvoicesView.jsx` (تمرير صحيح).
- `useAlWaseetInvoices.js` (الفلترة بـ `viewerUserId` تعمل أصلاً).
- قاعدة البيانات / RLS / المزامنة / إنشاء الفواتير.

## ملاحظات أمان

الفلترة على مستوى الواجهة تكفي هنا لأن:
- الفواتير نفسها تخضع لـ RLS (المدير العام يرى الكل، الموظف يرى فواتيره فقط — منطق سابق).
- الطلبات المحلية في `orders` تخضع لـ RLS أيضاً، فحتى لو حاول العميل تجاوز الفلتر، الاستعلام يعيد فقط ما تسمح به السياسات.

التغيير الحالي يضيف طبقة عرض إضافية تعزل بصرياً ما يخص كل مستخدم داخل الحساب المشترك، دون التأثير على المزامنة أو الربط أو الأرباح.
