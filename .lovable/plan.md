السبب المنطقي للخطأ

- الفاتورة 3406747 محفوظة محلياً بعدد 49 طلب حسب شركة التوصيل، وهذا صحيح في جدول delivery_invoices.
- داخل delivery_invoice_orders محفوظ فقط 14 طلب، وليس 49.
- من الـ 14 طلب المحفوظة، طلب واحد فقط مرتبط محلياً؛ لأن أغلب الصفوف المحفوظة لا تحتوي tracking_number أو qr_id داخل raw، والربط الحالي لا يقبل التخمين الخطر.
- يوجد خطأ مباشر في المنطق الحالي: عند فتح تفاصيل الفاتورة أو تنفيذ المزامنة الموجهة، إذا رجع API أي عدد من الطلبات مثل 14، يتم تحديث orders_last_synced_at كأن الفاتورة اكتملت، حتى لو المتوقع 49. لهذا النظام توقف عن إعادة المحاولة واعتبر الكاش مكتمل ظاهرياً.
- كذلك self-healing في الواجهة يعمل فقط إذا كان جدول طلبات الفاتورة فارغاً بالكامل، لكنه لا يعمل إذا كان ناقصاً جزئياً مثل 14/49.
- النتيجة: الفاتورة نفسها صحيحة 49، لكن تفاصيلها توقفت عند snapshot ناقص 14، ثم الربط لم يجد إلا طلباً واحداً آمناً للمطابقة.

خطة الإصلاح

1. إصلاح شكل الإشعارات كما كان
- إعادة أبعاد بطاقة الإشعار لتبقى داخل نافذة الإشعارات بدون تمدد يخفي الأطراف.
- إعادة الظرف/الطرف الملون لإشعار الطلب الذكي بشكل واضح مثل السابق، لكن بتدرج أنيق داخل نفس عرض الحد الجانبي فقط.
- إزالة سبب تضييق عنوان الإشعار: لا يكون الحد المتحرك طبقة تأخذ مساحة أو تؤثر على flex/layout.
- إبقاء النقطة الدائرية قرب وقت الإشعار كما في الصورة، وليس قرب العنوان.

2. إصلاح اكتمال كاش الفاتورة
- تعديل useAlWaseetInvoices.fetchInvoiceOrders بحيث لا يحدّث orders_last_synced_at إلا إذا cached_count >= orders_count.
- عند وجود كاش جزئي مثل 14/49، يتم استدعاء smart-invoice-sync بوضع targeted فوراً، وليس فقط عندما يكون الكاش صفراً.
- بعد self-heal الموجه، تعاد قراءة delivery_invoice_orders مباشرة حتى تظهر الزيادة في نفس نافذة الفاتورة.

3. إصلاح smart-invoice-sync للفاتورة الموجهة
- في target_invoice_external_id، لا يتم تحديث orders_last_synced_at إذا رجع API عدداً أقل من المتوقع.
- إذا رجع API 14 والعداد 49، تبقى الفاتورة incomplete وتظهر بوضوح في الواجهة وتدخل إعادة المحاولة القادمة.
- تشغيل retries داخل target mode مثل المسار العام، مع فواصل آمنة، حتى لا يقف عند snapshot ناقص.

4. تقوية الربط المحلي بدون تخمين خطر
- الاستمرار بالمطابقة الآمنة فقط عبر tracking_number / qr_id / delivery_partner_order_id / raw.id مع نفس الشركة ونفس حساب التوصيل.
- عند نقص tracking/qr في raw، نحاول إثراء البيانات من merchant-orders index قبل الحفظ.
- لا نربط بالهاتف أو السعر وحدهما لأن هذا قد يربط طلبات خاطئة، وهذا يخالف قاعدة zero error.

5. تنظيف حالة الفاتورة 3406747 والفواتير الناقصة
- إعادة ضبط orders_last_synced_at للفواتير التي cached_orders < orders_count، ومنها 3406747.
- إعادة تشغيل link_invoice_orders_to_orders بعد حفظ البيانات المثرية.
- عرض حالة واضحة في نافذة الفاتورة: المتوقع 49، المحفوظ 14 أو أكثر، المرتبط محلياً، والسبب إذا بقيت شركة التوصيل ترجع snapshot ناقص.

النتيجة المتوقعة

- الإشعارات تعود بحجمها الطبيعي داخل النافذة مع طرف ذكي ملون غير نشاز.
- الفاتورة 3406747 لن تُعتبر مكتملة إلا عند حفظ 49 طلب فعلاً.
- الربط المحلي سيكون دقيقاً 100% للطلبات التي تملك معرفاً مطابقاً محفوظاً؛ وأي نقص يبقى معلناً كـ incomplete بدلاً من إخفائه.
