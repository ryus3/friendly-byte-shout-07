## 1) إجاباتي على أسئلتك (شرح، بدون كود)

### أ) هل المزامنة والحذف التلقائي يعملان 100%؟
نعم، الآلية الحالية في `sync-order-updates` صحيحة ومحمية بثلاث طبقات:

1. **توكن لكل مستخدم لكل شركة**: الدالة تجلب جميع التوكنات النشطة من `delivery_partner_tokens` (الوسيط + مدن) وتستعلم API كل شركة بتوكنها الخاص. الطلب يُربط بصاحبه عبر `delivery_account_used`.
2. **حماية ضد فشل API (Set successfulFetches)**: لا يُحتسب الطلب "مفقوداً" إلا إذا كان الجلب من الشريك+الحساب نفسه ناجحاً. لو API الوسيط رجع 500 → لا يُلمَس أي طلب من الوسيط.
3. **2-Strike**: الطلب يجب أن يغيب مرتين متتاليتين (في جلستَي مزامنة ناجحتين) قبل أي إجراء.

ثم فحص محاسبي قبل الحذف: `receipt_received` + `delivery_invoice_orders` + `profits.status IN (settled, invoice_received)`. إن خلا → **حذف فعلي** (الـ trigger يُحرر المخزون). إن وُجد ارتباط مالي → **حذف ناعم** (status=cancelled, delivery_status=32).

→ **الاستنتاج**: لا توجد حالة يُحذف فيها طلب لشركة "أ" بسبب فشل API لشركة "ب"، ولا حالة يُحذف فيها طلب له ارتباط مالي.

### ب) لماذا "تم التسليم = 2" مع أن لا توجد طلبات نشطة؟
ليس خطأً. "تم التسليم" يعرض الطلبات بـ `delivery_status='4'` التي **لم تُؤرشف بعد**. هذه طلبات سُلِّمت فعلاً لكن لم تكتمل قاعدة الإكمال (Completion Gate = `delivery_status=4` AND `receipt_received=true`). يعني فاتورتها لم تُستلم بعد من الوسيط/مدن، فتبقى ظاهرة في "تم التسليم" حتى تُستلم الفاتورة فتنتقل إلى الأرشيف (338).

→ **الاستنتاج**: العدد صحيح. إذا أردتَ إخفاءها يجب تغيير منطق العمل (لا أنصح به لأن هذا هو الـ Completion Gate المعتمد).

### ج) كيف أضيف شركة كاملة مثل الوسيط بنفسي؟
نعم تستطيع، لكن النظام هجين بقصد:

| المستوى | يكفي لـ | تفعله بنفسك؟ |
|---|---|---|
| **Registry** (النموذج الحالي) | شركات API REST + JSON + Token قياسي (~80% من شركات العراق) | ✅ نعم بالكامل من الواجهة |
| **Adapter** (كود في `src/lib/adapters/`) | شركات بـ HMAC، OAuth، XML، أو شكل بيانات غير قياسي | ❌ تحتاج المطور |

**خطوات الإضافة بنفسك (Registry فقط):**
1. اطلب من الشركة وثيقة API (PDF/رابط).
2. افتح "إدارة شركة التوصيل" → "تسجيل شركة جديدة" → املأ الأقسام الستة.
3. سجّل دخولك بحساب الشركة من نفس النافذة.
4. شغّل مزامنة المدن/المناطق/الأحجام.
5. الشركة تظهر تلقائياً في الطلب السريع وقائمة الطلبات.

**متى ترسل لي وثيقة الشركة؟** فقط عندما:
- تستخدم HMAC أو توقيع رقمي.
- OAuth 2.0 بـ refresh token معقد.
- ترجع XML بدل JSON.
- شكل بيانات الإنشاء غير قياسي (مثلاً تحتاج تحويل مصفوفة منتجات إلى string).

في هذه الحالات أكتب Adapter صغير في 30 سطر يحوّل البيانات.

### د) فحص الهيكلية العالمية (تأكيد سريع)
- ✅ `delivery_partners_registry` يفصل الـ built-in عن المضاف.
- ✅ `delivery_partner_tokens` بـ partner_name + user_id (متعدد المستخدمين، متعدد الشركات).
- ✅ `city_delivery_mappings` + `region_delivery_mappings` كاش موحّد لكل شريك.
- ✅ Triggers الـ DB هي المصدر الوحيد للتغييرات على المخزون والكاش (مذكور في الذاكرة).
- ✅ `sync-order-updates` يدعم الشريكين بنفس المسار، 2-strike، فحص مالي.

→ **الهيكلية عالمية وصحيحة**. لا أنصح بأي إعادة هيكلة الآن.

### هـ) "أريده تطبيقاً خفيفاً جداً"
لاحظت أن سياسات الأداء مطبّقة (devLog، lazy loading للمكتبات الثقيلة). لو تشعر بثقل في صفحات معينة، أخبرني بالصفحة بالضبط وأفحصها لاحقاً. الآن لا يوجد دليل على ثقل في الكود الذي راجعته.

## 2) الإصلاحات الثلاث الفعلية في هذه الجولة

### إصلاح 1: زر "الرئيسية" — الصعود للأعلى دائماً
**الموجود الآن في `src/components/BottomNav.jsx`:** المنطق سليم، لكن `setTimeout(100)` أحياناً يفشل لو الصفحة أبطأ في التركيب. سأبسطه:
- في نفس الصفحة → `scrollTo({top:0, behavior:'smooth'})`.
- في صفحة أخرى → `navigate(target)` ثم scroll فوري + smooth fallback.

→ **سلوك مضمون: ضغطة الرئيسية = صعود دائماً، حتى وأنت في الرئيسية.**

### إصلاح 2: لمسة شريط الحالة (iOS/Android) → الصعود للأعلى
هذا مدعوم native في iOS Safari تلقائياً للـ `body` فقط، لكنه **لا يعمل** عندما يكون الـ scroll داخل `<main>` أو حاوية ابن. سأضيف **مستمعاً عالمياً** في `src/main.jsx` (أو Layout) يلتقط حدث `statusTap` (المُحاكى عبر تتبع `scrollY` + إيماءة) ويستدعي `window.scrollTo(0,0)` على كل الحاويات القابلة للسكرول.

طريقة موثوقة عالمية:
- أضع listener على `'click'` لعنصر فارغ `<div id="ios-status-tap-target" />` بأعلى 20px من الشاشة.
- بديل: استخدام `safe-area-inset-top` + onClick على الـ header العلوي فقط.

→ سأطبّق الحل الأبسط والأكثر استقراراً: عند نقر منطقة الـ header العلوية (`.app-header`)، يصعد للأعلى. هذا متوافق مع iOS وAndroid وكل المتصفحات.

### إصلاح 3: "تسجيل شركة جديدة" قابل للطي/الفتح
في `src/components/AddDeliveryPartnerSection.jsx`، الكارد `<Card>...تسجيل شركة توصيل جديدة...</Card>` ظاهر دائماً. سألفّه داخل `<Accordion type="single" collapsible>` مع `AccordionItem` افتراضي مغلق (collapsed). بهذا تُفتح فقط عند الحاجة، توفر شاشة، وتُحسّن السرعة الإدراكية.
