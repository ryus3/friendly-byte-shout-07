## التشخيص الدقيق

نعم، المشكلة أصبحت واضحة من فحص الكود والبيانات الحالية، وليست حظر من شركة التوصيل فقط.

### ما الذي خرب؟

1. **الفاتورة المشتركة تُحفظ مرة واحدة فقط** بسبب المفتاح الفريد:
   `delivery_invoices(external_id, partner)`
   لذلك فاتورة مثل `3319023` محفوظة باسم مالك واحد فقط: المدير العام، ولا تظهر للموظف إلا إذا وُجدت روابط طلبات محلية صحيحة داخل `delivery_invoice_orders`.

2. **صفحة متابعة الموظفين تعتمد الآن على `owner_user_id` للفواتير** أكثر مما تعتمد على طلبات الموظف المحلية المرتبطة. هذا يفسر لماذا اختفت الفواتير من متابعة الموظفين: الفاتورة المشتركة مالكها المدير، لكن المفروض تظهر أيضاً تحت الموظف إذا داخلها طلبات محلية للموظف.

3. **الربط نفسه ناقص حالياً**:
   - الفاتورة `3319023`: عددها في الفاتورة `45`، لكن جدول `delivery_invoice_orders` يحتوي `41` فقط، و`linked_rows = 0`.
   - الطلب `140872240` موجود محلياً ومسلّم، لكن غير موجود ضمن طلبات الفاتورة المخزنة، لذلك لا يستطيع الربط ربطه أصلاً.
   - الفاتورة `3343958` فيها طلب `140334640` في `delivery_invoice_orders` لكنه غير مرتبط محلياً، وهذا يدل أن دالة الربط تحتاج تقوية/تصحيح وتطبيقها على البيانات.

4. **أخطاء شركة التوصيل تظهر لأن النظام يحاول جلب تفاصيل الفواتير مباشرة من API عند فتح الصفحة أو التفاصيل** بدل أن يعرض الكاش المحفوظ فوراً. وعند رجوع `errNum:21` من الوسيط، الكود الحالي يعامله أحياناً كـ “لا فواتير” وأحياناً كخطأ، فيخفي المشكلة الحقيقية ويترك الكاش ناقصاً.

5. حسب توثيق الوسيط الذي أرسلته: **واجهات الفواتير تقبل Merchant token فقط**. لذلك إذا كان التوكن المخزن هو Merchant user token فسيعود `errNum:21`. هذا ليس حظر من الشركة، بل نوع توكن غير مناسب لواجهات الفواتير. يجب أن نميّز هذا بوضوح ونستخدم التوكن الصحيح أو نجدد التوكن من بيانات الحساب المحفوظة.

6. **موافقة الطلب الذكي** تستخدم الحساب المختار من قاعدة البيانات، وهذا جيد، لكن عند فشل التوكن أو fallback جلب رقم التتبع تظهر أخطاء جلسة الوسيط. يجب إضافة تجديد/تحقق للتوكن قبل الإنشاء، وعدم الاعتماد على حالة `activePartner` أو `localStorage`.

## خطة الإصلاح الجذرية

### 1. إصلاح بنية عرض الفواتير المشتركة بدون تكرار خاطئ
- نبقي الفاتورة محفوظة مرة واحدة في `delivery_invoices` حسب `(external_id, partner)`.
- نغيّر منطق العرض بحيث تكون “فاتورة الموظف” هي:
  - فاتورة فيها طلبات محلية مرتبطة أنشأها هذا الموظف، أو
  - فاتورة لحساب شركة توصيل يملكه/مسجل به الموظف ضمن صلاحيات المدير.
- في متابعة الموظفين:
  - المدير يرى فواتير الموظفين الذين يشرف عليهم/يديرهم فقط.
  - مدير القسم يرى موظفيه فقط.
  - الفاتورة المشتركة مثل `alshmry94` تظهر للمدير في تبويب فواتيره، وتظهر أيضاً في متابعة الموظفين للموظف المعني إذا تحتوي طلباته المحلية.

### 2. إصلاح الربط التلقائي للفواتير مع الطلبات المحلية
- تحديث دالة `link_invoice_orders_to_orders` لتطابق الطلب المحلي عبر كل المفاتيح بشكل صارم:
  - `tracking_number`
  - `delivery_partner_order_id`
  - `qr_id`
  - `delivery_invoice_orders.external_order_id`
  - `raw.id`
  - `raw.qr_id`
- إزالة السبب الذي يمنع ربط الطلبات المسلّمة الصحيحة.
- بعد الربط:
  - تحديث `orders.delivery_partner_invoice_id` تلقائياً.
  - إذا الفاتورة مستلمة فعلاً، تحديث `receipt_received` عبر التريگر الصحيح فقط، حتى يدخل الإيراد لقاصة مالك المنتج حسب قاعدة النظام.

### 3. إصلاح مزامنة تفاصيل الفاتورة والكاش
- عند دخول صفحة الفواتير أو متابعة الموظفين:
  - عرض بيانات قاعدة البيانات فوراً بدون انتظار API.
  - عدم إظهار خطأ شركة التوصيل إذا توجد بيانات محفوظة.
- الفواتير المستلمة والمكتملة الكاش لا تُجلب من شركة التوصيل كل مرة.
- الفواتير المعلقة أو الناقصة فقط تدخل مزامنة خلفية هادئة.
- عند فتح تفاصيل فاتورة ناقصة الطلبات:
  - تشغيل self-healing موجه لهذه الفاتورة فقط.
  - اختيار التوكن حسب `account_username + partner` وليس فقط `owner_user_id`، حتى تعمل الفاتورة المشتركة بين المدير والموظف.

### 4. إصلاح التعامل مع `errNum:21` في فواتير الوسيط
- في `smart-invoice-sync` و`alwaseet-api`:
  - `errNum:21` في واجهات الفواتير لا يمسح الكاش ولا يخفي الفواتير.
  - يتم تسجيله كـ `invoice_token_not_merchant` أو `invoice_endpoint_rejected` مع بقاء بيانات قاعدة البيانات ظاهرة.
  - إذا بيانات الحساب محفوظة، نحاول تجديد التوكن مرة واحدة ثم إعادة جلب تفاصيل الفاتورة.
- لا نرسل toast “تعذر الجلب من شركة التوصيل” عند الدخول العادي إذا عندنا كاش.

### 5. إصلاح صفحة متابعة الموظفين
- استبدال جلب الفواتير المباشر من `delivery_invoices.owner_user_id` بمنطق يعتمد على:
  - الموظفين المسموحين للمدير/مدير القسم.
  - `delivery_invoice_orders.order_id -> orders.created_by`.
  - حسابات التوصيل النشطة للموظفين عند الحاجة.
- عرض الفاتورة تحت كل موظف حسب طلباته المحلية فقط، وليس حسب مالك صف الفاتورة فقط.
- في تفاصيل الفاتورة داخل متابعة الموظفين، عرض الطلبات المحلية الخاصة بالموظف/النطاق الحالي.

### 6. إصلاح موافقة الطلب الذكي
- قبل إنشاء طلب في الوسيط/مدن:
  - اختيار توكن الحساب المحدد من `delivery_partner_tokens` مباشرة حسب المستخدم + الشركة + الحساب.
  - إذا التوكن منتهي أو مرفوض، تجديده من بيانات الحساب المحفوظة قبل محاولة الإنشاء.
  - عدم استخدام توكن شركة مختلفة أو توكن من `localStorage` بالخطأ.
- بعد إنشاء الطلب:
  - حفظ `tracking_number`, `qr_id`, `delivery_partner_order_id`, `delivery_account_used` بدقة في الطلب المحلي.
- الحفاظ على نظام “هل تقصد” الحالي للمدينة/المنطقة عبر `resolve_partner_location` وإرسال external IDs الصحيحة للشركة المختارة.

### 7. التحقق بعد التنفيذ
- تشغيل إصلاح موجه للفواتير:
  - `3319023`
  - `3343958`
- التأكد من:
  - عدد `delivery_invoice_orders` يطابق `orders_count` أو يظهر نقص واضح من API بدون مسح الكاش.
  - `linked_rows` يصبح أكبر من صفر للطلبات المحلية الموجودة.
  - الطلب `140872240` يتم ربطه إذا عاد فعلاً من API ضمن الفاتورة الصحيحة، أو يظهر تقرير أنه غير موجود في بيانات شركة التوصيل لتلك الفاتورة.
  - صفحة فواتير الطلبات تعرض الكاش فوراً.
  - متابعة الموظفين تعرض فواتير الموظفين المشرف عليهم.
  - لا تظهر أخطاء شركة التوصيل عند الدخول العادي للصفحة.
  - موافقة الطلب الذكي لا تفشل بسبب جلسة خاطئة إلا إذا الحساب نفسه يحتاج تسجيل دخول فعلي.
